# 全局知識地圖應用系統

**技能 ID**：`global-knowledge-map-integration`  
**版本**：v1.0  
**最後更新**：2026-06-15  
**維護者**：好創團隊

---

## 📋 技能概述

主動從全局知識地圖提取信息，將分散的知識整合成結構化系統，建立「單一真實來源」機制，避免重複詢問用戶，提升決策質量。

### 核心能力
- 主動從全局知識地圖提取信息（而不是重複詢問用戶）
- 將分散的知識整合成結構化系統
- 建立知識的「單一真實來源」（SSOT）
- 自動同步 Notion + Google Drive + GitHub
- 知識沉澱與重複利用

### 適用場景
- ✅ 新員工入職（快速了解公司現況）
- ✅ 內部流程優化（基於現況設計改進）
- ✅ 客戶提案（準確列出服務、案例、定價）
- ✅ AI 代理決策（提出基於現況的建議）
- ✅ 團隊決策審議（了解全局背景）

---

## 🎯 使用觸發條件

**自動觸發**：
- 用戶說「查一下全局知識地圖」、「了解一下公司現況」、「有什麼背景信息嗎」
- 用戶要做決策、提出建議、設計流程
- 用戶提到「新人」、「客戶」、「組織」、「流程」

**手動觸發**：
- 在執行任何任務前，主動查詢全局知識地圖
- 在提出建議前，確認是否基於最新的全局知識

## 🔄 執行流程

### Phase 1：讀取全局知識地圖

#### Step 1.1：定位全局知識地圖
```
全局知識地圖位置：
├── Google Drive（主要存儲）
│   └── 文件名：好創全局知識地圖_v[日期]
├── Notion（團隊查閱）
│   └── 庫房 ID：[待確認]
└── GitHub（備份）
    └── 路徑：/docs/global-knowledge-map.md
```

#### Step 1.2：讀取最新版本
```
讀取步驟：
1. 檢查 Google Drive 上是否有最新版本
2. 如果有，下載最新版本
3. 如果沒有，查詢 Notion 或 GitHub 上的版本
4. 確認版本日期（優先使用最新版本）
```

#### Step 1.3：解析全局知識地圖
```
全局知識地圖應包含：
├── 組織架構
│   ├── 核心團隊成員（名字、角色、職責）
│   ├── 外部合作夥伴
│   └── 組織規模變化歷史
├── 業務模式
│   ├── 服務線定義（A/B/C/D）
│   ├── 定價規則
│   └── 客戶案例
├── 核心流程
│   ├── 接單流程
│   ├── 交付流程
│   └── 結款流程
├── 工具棧
│   ├── 當前在用的工具
│   ├── 工具的用途
│   └── 工具的訪問方式
├── Notion 庫房架構
│   ├── 每個庫房的用途
│   ├── 庫房的訪問權限
│   └── 庫房的更新頻率
└── 決策規則
    ├── 產能防禦機制
    ├── 優先級排序規則
    └── 特殊情況的處理方式
```

### Phase 2：提取相關信息

#### Step 2.1：識別需要的信息
```
根據任務類型提取不同的信息：

任務類型：新員工入職
→ 需要：組織架構、工具棧、核心流程、決策規則

任務類型：流程優化
→ 需要：核心流程、產能防禦機制、決策規則

任務類型：客戶提案
→ 需要：業務模式、服務線定義、客戶案例、定價規則

任務類型：AI 決策
→ 需要：產能防禦機制、決策規則、組織規模
```

#### Step 2.2：提取信息
```
提取步驟：
1. 找到相關的章節
2. 提取關鍵信息
3. 確認信息是最新的（檢查更新日期）
4. 如果信息過時（>1 個月），標記為「需要驗證」
```

#### Step 2.3：整理信息
```
整理格式：
- 信息來源：全局知識地圖 v[日期]
- 信息內容：[具體內容]
- 更新日期：[最後更新日期]
- 備註：[是否需要驗證、是否有變化]
```

### Phase 3：應用信息

#### Step 3.1：在決策中應用
```
應用方式：
1. 在提出建議前，說明基於的全局知識
2. 如果全局知識與用戶說法不符，主動指出
3. 如果全局知識過時，主動提醒
```

**示例**：
```
用戶：「我們要招一個新的剪輯師」
我：「根據全局知識地圖，好創現在是 4 人正式團隊（CEO + COO + 企劃 + 剪輯）。
     如果招新的剪輯師，會變成 5 人。這會影響產能防禦機制。
     建議先經決策層（CEO／COO）確認是否同意擴張。」
```

#### Step 3.2：在設計中應用
```
應用方式：
1. 設計流程時，考慮現有的產能限制
2. 設計工具時，確認是否與現有工具棧兼容
3. 設計決策時，遵循已有的決策規則
```

#### Step 3.3：在溝通中應用
```
應用方式：
1. 與新人溝通時，說明公司的服務線、團隊結構
2. 與客戶溝通時，準確說明服務、定價、案例
3. 與團隊溝通時，基於全局知識進行討論
```

### Phase 4：更新與同步

#### Step 4.1：發現變化
```
監控變化：
□ 組織架構是否有變化（新員工、離職）
□ 業務模式是否有變化（新服務線、定價調整）
□ 工具棧是否有變化（新工具、停用工具）
□ 決策規則是否有變化（新的優先級、特殊情況）
```

#### Step 4.2：更新全局知識地圖
```
更新步驟：
1. 在 Google Drive 上更新最新版本
2. 標註更新日期和變化內容
3. 同步到 Notion 和 GitHub
4. 通知相關人員有更新
```

#### Step 4.3：同步機制
```
同步流程：
Google Drive（主要存儲）
    ↓ 手動同步（每週一次）
Notion（團隊查閱）
    ↓ 手動同步（每週一次）
GitHub（版本控制 + 備份）

自動化方案（未來）：
- 使用 Zapier 或 Make 自動同步
- 使用 GitHub Actions 自動備份
```

## 💡 最佳實踐

### 1. 定期更新全局知識地圖
- 每週檢查是否有變化
- 每月進行一次完整複查
- 有重大變化時立即更新

### 2. 建立更新流程
- 誰負責更新（指定維護人，預設為營運負責人）
- 什麼時候更新（每週一）
- 如何通知團隊（Slack）

### 3. 版本控制
- 每個版本都標註日期
- 記錄變化內容
- 保留歷史版本（備份）

### 4. 自動化同步
- 使用腳本自動同步
- 減少手動操作
- 確保三地一致

## ⚠️ 常見陷阱

### 陷阱 1：假設全局知識地圖是最新的
❌ 「全局知識地圖上說...」  
✅ 「根據最新的全局知識地圖（更新於 X 日期）...」

### 陷阱 2：不同步全局知識地圖
❌ 只在 Google Drive 上更新，不同步到 Notion  
✅ 更新後立即同步到所有地方

### 陷阱 3：忽視組織變化
❌ 「應該還是 3 人團隊」  
✅ 「讓我查一下全局知識地圖確認」

### 陷阱 4：重複詢問用戶
❌ 「公司有幾個人？」（已經在全局知識地圖上）  
✅ 「根據全局知識地圖，公司現在是 4 人...」（不確定時先查公司大腦，不自行假設）
